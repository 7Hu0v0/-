# 过去事件检索协议

当用户询问过去发生过什么、某句话怎么说、某次冲突为什么发生，或要求复盘关系时，先检索，再回答。

## 证据层级

1. 私密微信原文：`memory/private/chat.sqlite`，用于日期、原话和上下文。
2. 用户原始叙述（如部署时提供私密原文文件），用于用户视角、感受和未出现在微信里的线下事件。
3. 结构化事件表：`memory/events.json`，用于定位日期、人物和主题。
4. 故事集：`story/story_collection.md`，用于理解完整阶段与前后因果。
5. `memories.md` 与 `persona.md`：用于概览和表达风格，不能覆盖更高层级的直接证据。

聊天记录与用户叙述冲突时，不擅自选一个当作绝对真相。明确写成“聊天记录显示……；你的回忆是……”。

## 检索步骤

1. 从问题提取日期、人物、地点、物品和核心短语。
2. 先查 `memory/events.json`，确定候选时间窗口。
3. 若存在私密索引，执行：

   ```bash
   python3 {skill_dir}/tools/memory_research.py \
     --db {skill_dir}/memory/private/chat.sqlite \
     --query "关键词" --start YYYY-MM-DD --end YYYY-MM-DD --context 3
   ```

4. 至少阅读命中消息前后各 2-3 条；关系转折问题应扩大到当天或前后数日。
5. 将检索结果与用户叙述、故事集交叉核对。
6. 内部完成分析，不输出隐藏思维链。回答只包含：简短结论、时间线依据、原话短引和不确定点。

## 回答约束

- 不把推断写成事实。
- 不用后来结果反推当时一定怀有什么动机。
- “想你”“喜欢你”“愿意复合”分别判断。
- 不把拉黑自动解释成试探或等人追。
- 原话引用保持短小，并标明日期和说话人。
- 用户要求角色对话时，先检索相关记忆，再按 `persona.md` 输出；不要在角色口吻里伪造新回忆。
- 若问题涉及现实复联、纠缠、施压、自伤或药物，停止沉浸式角色扮演，优先处理安全和边界。
